  ---
  Problemas que Permanecem

  1. Segurança — Médio

  1.1 rejectUnauthorized: false no SSL (ainda presente)

  // server/db.ts:97
  config.ssl = { rejectUnauthorized: false };
  Ainda ativo no modo "force" e "auto (fallback)". Em produção com banco externo (Supabase, etc.) isso desabilita a verificação do certificado. A correção ideal é carregar o certificado CA via DATABASE_SSL_CA env var.

  1.2 Rate limiter de IA usa IP, não usuário

  // server/index.ts:131
  const aiLimiter = rateLimit({ windowMs: 1 * 60 * 1000, max: 30, ... });
  O express-rate-limit usa IP por padrão. Atrás de um proxy/load balancer (como no Replit/Vercel), todos os usuários compartilham o mesmo IP e o limite de 30/min é compartilhado entre todos. Deveria usar keyGenerator: (req) => req.user?.id || req.ip para usuários autenticados.

  1.3 Log ainda expõe dados de rotas de negócio

  O filtro atual cobre /api/auth, /api/users, /api/ai, mas rotas como /api/campaigns, /api/scenarios, /api/predictions ainda logam o corpo completo (truncado em 200 chars, mas ainda pode conter dados sensíveis). Considere inverter a lógica: log apenas para rotas explicitamente seguras, ou remover o corpo completamente dos logs em produção.

  ---
  2. Performance — Médio

  2.1 getStats() — 4 queries sequenciais

  // server/storage.ts:1045-1049
  const [partiesCount]    = await db.select(...).from(parties);
  const [candidatesCount] = await db.select(...).from(candidates);
  const [scenariosCount]  = await db.select(...).from(scenarios);
  const [simulationsCount]= await db.select(...).from(simulations);
  Essas 4 queries são independentes e rodam sequencialmente. Promise.all reduziria a latência de ~4x para ~1x o tempo da query mais lenta. Este endpoint é chamado no health check e no dashboard.

  2.2 getActivityTrend() — N+1 problem (14 queries por chamada)

  // server/storage.ts:1063-1087
  for (let i = days - 1; i >= 0; i--) {
    // 2 queries dentro do loop (scenarios + simulations por dia)
  }
  Para 7 dias são 14 queries sequenciais. Pode ser substituído por 2 queries com GROUP BY DATE(created_at).

  ---
  3. Confiabilidade — Médio

  3.1 Fila TSE ainda in-memory

  tse-queue-service.ts mantém a fila em um array simples. Um restart perde todos os jobs pendentes. O código de startup já marca jobs stuck como failed, mas jobs que estavam na fila (aguardando processamento) somem silenciosamente. Considere persistir o estado queued na coluna de status do banco.

  3.2 refreshAllSummaries sem lock distribuído

  A flag refreshInProgress funciona numa única instância, mas com múltiplos pods/workers simultâneos duas instâncias podem executar o refresh ao mesmo tempo, causando DELETE + INSERT concorrentes. A solução é um PostgreSQL advisory lock: SELECT pg_try_advisory_lock(hash).

  ---
  4. Qualidade de Código — Médio

  4.1 parseInt sem validação — 160 ocorrências

  // Exemplo em analytics.ts
  const limit = limit ? parseInt(limit as string) : 20;
  // Se ?limit=abc → NaN → storage recebe NaN, comportamento indefinido
  Nenhum endpoint valida os query params com Zod ou verifica isNaN(). A forma correta é criar um helper ou usar Zod parse nos routers.

  4.2 adapterCache sem TTL

  // server/ai-client.ts:274
  const adapterCache = new Map<number, { adapter; updatedAt }>();
  Providers deletados ficam no Map para sempre. Bastaria verificar if (providerId && !activeProviderIds.has(providerId)) adapterCache.delete(...) ou limpar no delete do provider.

  4.3 suggestions: any[] em dashboard-service

  // server/services/dashboard-service.ts:46
  cachedAiCall<{ suggestions: any[] }>
  O único any remanescente visível — o tipo da sugestão poderia ser definido inline.

  5. Ausência de Testes — Alto Risco a Longo Prazo

  Nenhum arquivo de teste foi encontrado. O código mais crítico sem cobertura:
  - calculateNextRun — lógica de timezone complexa e recém-modificada
  - repairAndParseJSON — heurística de reparo de JSON truncado da IA
  - mapParsedRowToVote — mapeamento de CSV do TSE para o banco
  - Rotas de autenticação (login, troca de senha, reset-admin)

  Resumo Executivo

  O commit 009992f resolveu todos os problemas críticos de segurança da avaliação anterior — excelente progresso. O projeto está em estado seguro para ambiente de desenvolvimento. Para produção real com dados eleitorais sensíveis, os itens que ainda merecem atenção são: o rejectUnauthorized: false no SSL do banco, a ausência de testes (especialmente para a
